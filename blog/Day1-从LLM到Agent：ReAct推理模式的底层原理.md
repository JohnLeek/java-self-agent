# 从LLM到Agent：ReAct推理模式的底层原理

> LLM是一个博学的"大脑"，但要让它成为能干活儿的Agent，还需要一个"手脚协调系统"。这篇文章带你理解Agent的核心引擎——ReAct推理循环的底层运作机制。

---

## 一、LLM的"诅咒"：知道一切，却做不了任何事

GPT-4能通过律师资格考试，但它不能帮你发一封邮件。DeepSeek能写出漂亮的代码，但它不能帮你把代码部署到服务器。

这不是能力问题，是**接口问题**。LLM本质上是一个函数：

```
String → String
```

输入文字，输出文字。它没有手、没有键盘、没有网络——它被困在文字的世界里。

让LLM突破这个限制的关键思路是：**给LLM一套"工具"，让它学会自己决定什么时候用什么工具**。这就是Agent的起点。

## 二、ReAct：当推理遇上行动

2022年，Google和普林斯顿的论文《ReAct: Synergizing Reasoning and Acting in Language Models》提出了一个简洁但强大的范式：

```
传统LLM: 问题 → 回答（一步到位）
ReAct:   问题 → 思考 → 行动 → 观察 → 思考 → 行动 → ... → 回答
```

这个循环的精妙之处在于，它把"思考"和"行动"耦合在一起：

- **Reasoning（推理）**：LLM分析当前状态，输出"我应该做什么"的思考过程
- **Acting（行动）**：执行具体的工具调用（搜索、计算、调用API）
- **Observation（观察）**：工具返回结果，成为下一轮推理的输入

这个循环的本质是什么？是**把不确定性转化为可迭代的收敛过程**。LLM不需要在第一次就给出正确答案——它可以通过多步推理和工具调用，逐步逼近正确答案。

### 为什么ReAct比"一次调用"更可靠？

用一个例子来说明。问LLM："2024年诺贝尔物理学奖得主是谁？"

```
纯LLM（无ReAct）：
  → 训练数据截止2023年，不知道 → 要么拒答，要么"编造"一个答案（幻觉）

ReAct Agent：
  → Think: 我不知道，需要搜索最新信息
  → Action: search("2024 诺贝尔物理学奖")
  → Observation: 约翰·霍普菲尔德和杰弗里·辛顿
  → Think: 信息足够，可以回答了
  → Answer: 2024年诺贝尔物理学奖授予...
```

ReAct用"承认不知道+主动获取信息"替代了LLM的"硬猜"。这就是Agent比纯LLM可靠的根本原因。

## 三、Function Calling：ReAct的工程实现

ReAct是一个理论范式，工程上如何实现？答案就是各大LLM厂商的Function Calling机制。

从消息序列的角度理解Function Calling：

```
第1轮:  System Prompt → "你是助手，可以使用以下工具..."
        User → "帮我算3.14*256"
        Assistant → { "tool_calls": [{ "name": "calculate", "args": { "expr": "3.14*256" }}] }

第2轮:  System Prompt（同上）
        User → "帮我算3.14*256"
        Assistant → { "tool_calls": [...] }
        Tool → { "result": "803.84" }  ← 工具执行结果追加到消息历史
        Assistant → "3.14×256的结果是803.84"
```

关键点：**工具调用结果不是通过"回调"返回的，而是通过追加到消息历史中**。LLM看到上下文里多了一条工具返回的消息，自然地基于此继续推理。

这个设计的精妙之处在于——**LLM本身不需要任何架构改造**。它只是看到消息历史里多了工具调用和工具返回，然后继续做它最擅长的事：给定上下文，预测下一个token。

## 四、System Prompt：Agent的"宪法"

如果说ReAct循环是Agent的引擎，那System Prompt就是Agent的"行为准则"。同样的工具集，不同的System Prompt，Agent表现可能天差地别。

### 为什么System Prompt如此重要？

LLM是"概率性"的。面对用户"帮我算一下"，它可以：
- 直接心算回答（可能对，可能错）
- 调用calculate工具（更可靠）
- 调用search工具（完全错误的工具选择）

System Prompt的作用就是**改变这几个选项的概率分布**。好的System Prompt让"调用正确的工具"成为最高概率的选择。

### 一个反直觉的事实

很多人认为Agent开发的难点在"写代码"，实际上**70%的工作量在调Prompt**。代码只是搭建了"手脚"（工具），System Prompt决定了Agent怎么用这些手脚。

比如你写了一个计算器工具，但Agent总是心算而不是调用工具。这不是代码Bug，是Prompt设计问题——你没有在Prompt中明确告诉它"**必须**使用工具而不是心算"。

## 五、Single Agent vs Multi Agent：两种架构哲学

Agent架构有一个根本性选择：一个全能的Agent，还是多个分工的Agent？

### Single Agent：极简主义

```
用户 → 一个ChatClient（拥有所有工具）→ 回答
```

优势：简单。没有调度开销，不需要协调。适合工具数量少（<10个）、任务边界清晰的场景。

劣势：上下文压力。所有工具的JSON Schema都塞在System Prompt里，工具越多LLM选择越困难。就像一个工程师面前摆了50个工具箱，每次都重新判断用哪一个。

### Multi Agent：分工协作

```
用户 → Orchestrator（规划）→ 调度
         → Search Agent（只有搜索工具）
         → Tool Agent（只有执行工具）
         → Memory Agent（只有记忆工具）
       → Orchestrator（汇总）→ 回答
```

优势：每个Agent只面对一个"精简的工具列表"，选择准确率高。且职责隔离——搜索Agent不会不小心删文件，因为它的工具列表里根本没有删除工具。

劣势：多了一跳LLM调用，延迟增加。规划可能出错（Orchestrator选错了Agent）。

### 如何选择？

工具<10个、任务简单 → Single Agent。工具>15个、需要职责隔离、需要可视化中间过程 → Multi Agent。

## 六、关于"Agent门槛"的思考

很多人觉得Agent开发门槛高，其实Agent的本质只有三个要素：

1. **LLM**（大脑）—— 负责推理
2. **工具**（手脚）—— 负责执行
3. **循环**（协调）—— 负责迭代

这三者加起来不到100行代码的核心逻辑，就能实现一个基础的ReAct Agent。Spring AI这样的框架把这个过程封装得更优雅，但底层的原理不会变。

理解了这个"铁三角"，你就理解了所有Agent框架的本质——它们只是在这个基本循环上加了各种工程优化：记忆管理、并发控制、可观测性、安全审批...

## 小结

ReAct推理模式是Agent的基石。它的核心思想是：**让LLM在推理-行动-观察的循环中迭代，而不是要求一次给出正确答案**。这个简单的范式，加上System Prompt的行为引导，构成了所有现代Agent系统的基础。

Function Calling是ReAct的工程落地——通过在消息序列中追加工具调用和工具返回，让LLM自然地基于工具结果继续推理，而不需要对LLM做任何架构改造。

理解了这个基础，后续的Skill系统、MCP协议、多Agent编排，都是在同一个地基上搭建的更高层建筑。
