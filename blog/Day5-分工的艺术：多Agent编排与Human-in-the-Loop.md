# 分工的艺术：多Agent编排与Human-in-the-Loop

> 一个全能Agent就像一个人扛所有活儿——什么都能干，什么都干不精。这篇聊两个核心命题：怎么把任务分给多个专业Agent协作？以及，什么时候Agent应该自己决定，什么时候应该等人类点头？

---

## 一、从"单体"到"微服务"：Agent架构的必然演进

软件架构的演进历史有一个清晰的主线：**单体→分层→微服务**。Agent架构也在走同样的路。

单Agent模式（一个LLM + 所有工具）在工具数量少、任务简单时是完美的。但随着系统复杂度增长，单Agent面临三个瓶颈：

**上下文瓶颈**：每个工具的JSON Schema都挤在System Prompt里。20个工具可能消耗5000+ tokens仅用于工具描述，挤占了推理和回答的空间。

**注意力瓶颈**：LLM面对太多工具选择时，准确率下降。就像给你50个工具箱让你快速找到需要的那个——你知道每个箱子的用途，但选择本身就要消耗时间。

**安全瓶颈**：一个拥有"读文件"和"删文件"两种工具的Agent，可能在"读文件"任务中意外触发"删文件"——因为它们全在一个工具列表里，LLM可能混淆。

多Agent架构解决这三个瓶颈的核心思路很简单：**给每个Agent只分配它需要的工具，不让它看到不需要的。**

## 二、Orchestrator模式：谁来做"项目经理"？

多Agent协作有多种组织模式。最灵活的是"自由市场"模式（LLM之间自由通信），但可控性最差。我们选择的是**Orchestrator模式**——一个"项目经理"调度几个"专家"。

### 为什么需要Orchestrator？

```
自由市场模式：
  搜索Agent → 工具Agent → 搜索Agent → 记忆Agent → ...
  任意跳转，没有统一的调步者

Orchestrator模式：
  Orchestrator → 搜索Agent → 回来
  Orchestrator → 工具Agent → 回来
  Orchestrator → 汇总 → 输出
  每一步都在Orchestrator的掌控中
```

自由市场的问题是**不可预测**。你不知道Agent们会互相调多少轮，Token消耗难以控制，出了问题无法定位是哪个Agent。Orchestrator模式把控制权交还给开发者。

### Orchestrator的核心职责不是"调度"，而是"判断"

很多人以为Orchestrator的工作是"把任务分发给Agent"。其实这只是表面。Orchestrator真正的核心职责是**判断哪个Agent适合处理当前任务**。

这是一个分类问题。因为只需要分类（而不是生成），所以Planner可以用更小的模型。我们用的是同一个 `deepseek-chat` 模型，但Prompt极简：

```
判断需要哪些专家，只输出标签（逗号分隔）。
SEARCH(查知识库) TOOL(计算/翻译/脚本) MEMORY(笔记/用户)。
都不需要输出 NONE。
```

Planner只输出几个关键词，不需要长篇大论。这让Planner的调用非常快（通常在0.5秒内完成）。

## 三、专家隔离：为什么"看不见"比"看得见"更重要？

每个Specialist有独立的System Prompt和工具集。这个设计的精妙之处不在"给每个专家配置了合适的工具"，而在于**阻止每个专家看到它不应该看到的工具**。

```
搜索专家看到：{ search, mcp_read_file, mcp_fetch }
工具专家看到：{ calculate, translate, rollDice, mcp_write_file }
记忆专家看到：{ noteToSelf, readMyNotes, getUserProfile }
```

工具专家根本不知道 `search` 工具的存在——所以它绝不可能在正确应该搜索的时候，错误地调用了一个搜索工具。这种"不知道"比任何规则约束都更可靠。

### 这和数据库权限的最小权限原则相同

数据库用户只被授予它需要的权限。Agent的"权限"就是"可见的工具集"。搜索Agent不需要写文件，所以它的System Prompt里根本没有文件写入工具的Schema——LLM根本不知道这个功能的存在。

这是一个根本性的安全改进：**不是告诉Agent"你不能调用这个工具"（软约束），而是让Agent根本不知道这个工具存在（硬约束）。**

## 四、Human-in-the-Loop：人和机器的协作边界

Agent可以自动完成大部分操作，但有些操作必须由人类确认。划分这个边界是一个重要的设计问题。

### 什么操作需要审批？

一个简单的划分标准是**可逆性**：

- 读取操作（读文件、搜索）→ 可逆，自动执行
- 幂等操作（创建不存在的文件）→ 基本可逆，低风险
- 破坏性操作（删除文件、执行脚本、外部请求）→ 不可逆，必须审批

但可逆性不是唯一标准。外部请求（比如调用付费API）即使在技术上可逆（花钱了但获取到信息就不算浪费？），也应该审批——因为涉及成本。

### 审批流的工程实现：为什么用CompletableFuture？

审批流的技术难点是**一个HTTP请求需要"等待"另一个HTTP请求**。

```
对话请求 → Agent推理 → 需要审批 → ??? → 用户在前端点确认 → Agent继续
```

传统的做法是拆成两个独立的API调用：Agent先返回"需要审批"的状态，前端展示确认框，用户确认后前端再发起一个新请求让Agent继续。但这种方案把状态管理推给了前端，增加了复杂度。

我们选择了更简洁的方案：**服务端用CompletableFuture阻塞等待**。

对话请求的处理线程在遇到审批点时，创建一个 `CompletableFuture`，然后 `.join()` 阻塞等待。前端确认后，另一个HTTP请求完成这个Future，阻塞的线程继续执行。配合60秒超时机制，不会无限阻塞。

这个方案的精妙之处在于：**对Agent的逻辑来说，审批就像调用一个"有点慢的工具"**——Agent调用了审批工具，等待了几秒（用户在确认），拿到了结果（通过或拒绝），然后继续。Agent完全不需要知道"等待人类确认"和"等待API返回"有什么区别。

## 五、支持中断：Agent不是非得执行完

一个容易被忽视但很重要的功能：**用户可以随时停止Agent执行**。

传统程序执行到一半很难优雅停止（需要处理事务回滚、资源释放等）。Agent的停止相对简单——在每个步骤前检查一个标志。

多Agent架构让中断更自然：每个Specialist调用前都是中断检查点。用户点击停止 → 设置中断标志 → Orchestrator在下一个Specialist调用前检测到 → 提前完成Flux流。

这个设计体现了**合作式而非抢占式的中断模型**。不是强行杀死线程（可能留下脏数据），而是在已知的安全点上检查、自行选择停止。这比硬杀线程优雅得多，也更安全。

## 六、SSE协议：为什么不是WebSocket？

前端和后端之间用SSE（Server-Sent Events）推送流式数据。很多人问：为什么不用WebSocket？

SSE是单向的（服务器→客户端），WebSocket是双向的。对于Agent对话这个场景，数据流是高度不对等的：

- 服务器→客户端：大量流式数据（每个思考步骤、每个文本片段、每轮追踪数据）
- 客户端→服务器：极少（用户发一条消息，偶尔点一次停止按钮）

这种"重下行、轻上行"的特征，SSE是更合适的选择。它比WebSocket更简单（标准HTTP、自动重连、不需要心跳），而且兼容所有HTTP基础设施（负载均衡、代理、CDN）。

WebSocket更适合需要高频双向通信的场景（比如协同编辑、游戏）。Agent对话用WebSocket是过度设计。

## 七、多Agent架构的代价

任何架构选择都有trade-off。多Agent架构的好处是可控性和隔离性，代价也很明显：

1. **延迟增加**：Planner一次调用 + 每个Specialist一次调用 + Summarizer一次调用。如果调用了3个专家，就是5次LLM调用，延迟是单Agent的5倍。

2. **规划可能出错**：Planner判断错了需要哪个专家，后面的执行就全偏了。单Agent没有这个问题（它自己决定每一步）。

3. **信息传递损失**：一个专家执行的结果传给Orchestrator，Orchestrator再传给另一个专家——每次传递都可能丢失细节。

所以多Agent不是"更好的架构"，而是"适合特定场景的架构"。任务边界清晰、需要职责隔离、需要可视化中间过程的场景，多Agent很合适。简单一问一答的场景，单Agent可能更好。

## 小结

多Agent架构的核心不是"调度"，而是**隔离**——让每个Agent只面对它需要的工具集，不让它看到不需要的。这是从根本上减少LLM的选择困难和安全风险。

Human-in-the-Loop的核心是**对不可逆操作的敬畏**——承认Agent会犯错，在犯错后果严重的环节引入人类判断。

这两个设计共同回答了一个问题：**如何让Agent既保持自主性，又不失去控制？**
