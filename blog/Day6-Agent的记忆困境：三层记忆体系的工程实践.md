# Agent的记忆困境：三层记忆体系的工程实践

> 一个没有记忆的Agent，每次对话都是"初次见面"。但"记忆"远不止"把聊天记录存下来"这么简单。这篇文章解剖Agent记忆系统的三个维度，以及每个维度背后的工程挑战。

---

## 一、LLM的无状态本质

LLM是无状态的。每一轮对话，所有历史消息都必须重新发送给它。你可能会问：为什么不能让它"记住"？

根本原因在于Transformer架构的工作原理。LLM的"记忆"不是在一个持久存储中读写的——它的"记忆"就是当前这个大矩阵（模型参数）乘以输入序列，产生输出。一旦输出完成，计算就结束了，状态就完全消失了。

这和人类大脑完全不同。人脑有海马体（短期记忆）和皮层（长期记忆），可以在不同时间尺度上存储和检索信息。LLM只有"当前上下文"这一个记忆载体。

所以，Agent的记忆系统本质上是**在LLM之外构建一个人工的"海马体+皮层"**，让Agent获得跨时间的信息保持能力。

## 二、为什么"把聊天记录全发给LLM"不是好方案？

直觉上最简单的"记忆"方案：每次对话把所有历史消息全发给LLM。

两个致命问题：

**1. Token成本。** 假设每轮对话平均2000 tokens，10轮就是20000，100轮就是200000。deepseek-chat每百万token的价格看起来不贵，但累积起来，每月仅"记历史"的token消耗就能上千元。

**2. 上下文窗口有限。** 即使是最新的模型（比如128K窗口），长上下文也面临"lost in the middle"问题——LLM对上下文开头和结尾的信息记得住，对中间部分的信息容易忽略。论文证明，当信息在上下文中间位置时，LLM的检索准确率显著下降。

所以，仅仅"记住"是不够的。需要的是**智能的记忆管理**——该记的记，该忘的忘，该压缩的压缩。

## 三、三层记忆体系的设计逻辑

我们的三层记忆不是随意分的。每一层服务于不同的时间尺度：

### 短期记忆：会话级（分钟~小时）

短期记忆的目标是**维护当前对话的连贯性**。它回答的问题是："刚才聊到哪里了？"

核心策略是**滑动窗口+摘要压缩**。最近的N条消息保留原文（因为它们是最相关的），较早的消息压缩成摘要（保留关键信息但不保留细节）。

为什么用Token数量而不是消息条数作为窗口大小？因为一条"好的"和一条2000字的文档，对上下文的影响是100倍。用消息条数，会导致不可预测的Token消耗。用Token预算，就能精确控制在上下文的50%以内。

### 长期记忆：用户级（天~月）

长期记忆的目标是**跨会话保持用户信息**。它回答的问题是："这个用户是谁？"

技术上，长期记忆是一个简单的**信息提取+检索**管道：从对话中提取关键事实（"我叫张三"→存储"姓名:张三"），下次对话时检索并注入System Prompt。

当前版本用关键词匹配提取（"我叫"、"我是"、"我喜欢"等触发词），这在80%的场景下够用。但更复杂的信息（如"我最近在做一个微服务项目，用的Spring Cloud，但网关性能不太好"），关键词匹配就不够了——需要LLM来做信息提取。

这就是为什么长期记忆的实现预留了升级空间：当前版本是关键词匹配，但架构支持替换为LLM提取（只需要改 `extractAndStore` 方法的实现）。

### 工作记忆：任务级（秒~分钟）

工作记忆的目标是**在执行复杂任务时暂存中间结果**。它回答的问题是："我分析到哪一步了？"

这个概念来自Anthropic的"Scratchpad"模式。人类在解决复杂问题时，会在草稿纸上记录中间步骤。Agent也需要同样的能力——不是所有信息都要记住永久，有些只是在当前任务中暂存。

工作记忆通过 `noteToSelf` 和 `readMyNotes` 两个工具暴露给Agent。Agent可以在分析完A项目后记一笔，分析完B项目后再记一笔，最后读取所有笔记做综合对比。

工作记忆存储在数据库中，不受Token窗口限制。这是一个关键特性——Agent可以在"脑中"只保留当前思考的那一步，而所有历史分析结果安全地存在外部，随时可以"查笔记"找回。

## 四、摘要压缩：信息论视角

摘要压缩是短期记忆中最关键的工程挑战。问题可以表述为：

```
给定一段对话历史（N条消息，总共T个tokens），
生成一个摘要（最多S个tokens），
使得摘要保留尽量多的关键信息。
```

这是信息压缩，不是无损编码。关键是**保留什么、丢弃什么**。

Anthropic的研究表明，好的摘要应该保留：
- 用户身份信息（谁在说话）
- 讨论主题和决策（在聊什么、决定了什么）
- 待办事项（接下来要做什么）

可以丢弃：
- 中间的推理过程（已经得出结论了）
- 失败的尝试（除非对后续有启发）
- 客套话和闲聊

我们用一个专门的LLM调用来做摘要，Prompt极简化：

```
你是对话摘要助手。从对话中提取：用户身份、偏好、讨论主题、重要决策、待办事项。只输出摘要，不超过200字。
```

200字是一个经验值——太少会丢失关键信息，太多则压缩效率低。

### 摘要合并策略

多次压缩后，摘要可能累积很多。我们的策略是**增量式合并**：新摘要用分号追加到旧摘要后面，而不是替换。

这个选择的考量是：旧摘要中的信息可能已经不在当前窗口中了，替换意味着丢弃那些信息。追加保留更完整的历史，代价是摘要越来越长。当摘要本身超过一定长度时，可以用同样的LLM调用对它再做一次压缩。

## 五、Token估算：精确 vs 简单

Token计数是记忆系统的基础操作。最精确的方式是用模型的tokenizer（如tiktoken），但引入额外的依赖和复杂度。

我们用了一个近似算法：

```
中文字符 ≈ 0.5 token/字（2字1 token）
英文字符 ≈ 0.25 token/字（4字1 token）
```

这个估算的误差在±20%左右。对于Token预算管理来说，这个精度完全够用——因为预算本身就是"上下文的50%"这种大致的比例，不需要个位数精度的token计数。

这个取舍体现了工程中的一个重要原则：**不需要100%精确的地方，不要去追求100%精确**。Token估算不是金融交易，±20%的误差不影响系统的正确性，但精准计数引入的复杂度是真实的。

## 六、"Lost in the Middle"问题

这是一个几乎所有长上下文应用都会遇到但很少有人解决的问题。

研究表明，LLM对上下文不同位置的注意力是不均匀的：

```
[开头] ← 注意力高
[中间] ← 注意力低 ← 信息容易被"遗忘"
[结尾] ← 注意力最高
```

所以简单地把所有历史消息按时间顺序排列，把最重要的信息放在中间位置——可能适得其反。

我们的缓解策略是两层：
1. 历史摘要放在开头（"充分利用高注意力区域"）
2. 最近的消息放在结尾（"最近的对话自然最相关"）

中间不需要放东西——被截断的旧消息已经被摘要覆盖了。

## 七、可观测性：Agent的"黑盒透视镜"

最后聊聊可观测性。Agent和传统程序最大的不同是：**它的大部分"思考"是不可见的**。

传统程序：调用函数A → 参数是X → 返回Y → 调用函数B... 每一步都可以打日志。
Agent程序：LLM看到了什么上下文？它为什么调了工具A而不是工具B？它思考了多少轮？

可观测性系统的核心是**把Agent的每一步决策都记录下来**——每次LLM调用（PLAN、AGENT_CALL、SUMMARY）、每次工具调用（search、calculate、noteToSelf），都记录耗时、输入、输出、状态。

我们用了Span模型来表达：每个步骤是一个Span，多个Span构成一个Trace。前端可视化后，用户看到的是：

```
[00:00.5] PLAN    Orchestrator判断需SEARCH+TOOL
[00:02.8] AGENT   搜索专家 检索知识库（调了search工具）
[00:03.0] AGENT   工具专家 计算结果（调了calculate工具）
[00:06.8] SUMMARY 汇总生成回答
总耗时 6.8s | 工具调用 3次
```

这个"调用链视图"对调试Agent行为极其重要——没有它，你根本不知道Agent调了什么、哪一步慢了、何时出错了。

### 一个反直觉的计数问题

工具调用计数看似简单，但在Spring AI的流式调用中有一个陷阱：`@Tool` 方法是在Spring AI内部线程中执行的，而前端接收到流式响应后读取计数器是在Controller的线程中——它们是不同的线程。

如果用 `ThreadLocal` 存计数器，两个线程隔离，Controller读不到工具调用的次数。所以必须用 `AtomicInteger`（线程间共享）——这是一个因为框架的内部线程模型而导致的设计决策。

## 小结

Agent记忆不是"把聊天记录存下来"那么简单。三项核心挑战：

1. **信息压缩**：怎么在Token预算内保留最多关键信息？（摘要压缩策略）
2. **跨时间检索**：怎么在需要的时候找到相关信息？（三层记忆分层存储+检索）
3. **注意力分配**：怎么让LLM注意到真正重要的信息？（摘要放Context开头 + 最近消息放结尾）

这三个挑战回答了同一个根本问题：**如何让一个本质上无状态的LLM，表现得像有记忆一样？**
