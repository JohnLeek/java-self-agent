# 工具抽象的艺术：可插拔Skill系统的设计哲学

> 如果Agent是一个操作系统，Skill就是它的App。这篇文章不教你写代码，而是带你理解一个关键问题：Agent的工具系统应该怎么设计，才能像手机App Store一样"安装即用"？

---

## 一、问题的起点：工具不是越多越好

一个常见的误解是：给Agent配的工具越多，它就越强大。

实际上，工具数量超过一定阈值后，Agent的表现反而会下降。原因有三层：

**第一层：JSON Schema污染。** 每个工具都需要在System Prompt中描述自己的参数格式（JSON Schema）。10个工具可能有5000+ tokens的Schema描述，挤占了本该用于思考和回答的上下文空间。

**第二层：选择困难。** LLM面对20个工具，要从中选一个最合适的——这本身就是一个"分类问题"。工具越多，分类错误率越高。

**第三层：名称冲突。** 不同来源的工具可能有相似名称或功能，LLM不知道选哪个。

所以，好的工具系统的首要目标不是"方便加工具"，而是**在添加工具的同时，不让Agent变笨**。

## 二、两层Skill架构的设计思考

我们把Skill分成两层：内置Skill和外部Skill。这个分层的背后是一个重要的设计判断：

### 内置Skill：信任域

计算器、翻译、骰子——这些是Agent的"原生能力"。它们有几个共同特征：
- **行为可预期**：代码是我们写的，我们知道每次调用的确切行为
- **不需要安全审查**：计算器不会删文件，翻译不会发网络请求
- **性能要求高**：这些工具会被频繁调用，不能有额外开销

所以内置Skill直接以Java代码的形式存在，用 `@Tool` 注解暴露。调用开销为零。

### 外部Skill：扩展域

外部Skill是用户自己导入的能力。它们的特征是：
- **来源不可信**：可能是第三方提供的，行为不保证
- **协议多样**：HTTP API、本地脚本、MCP协议...
- **需要安全管理**：脚本需要沙箱，HTTP需要超时，MCP需要进程管理

所以外部Skill存储在数据库中，每个都有独立的生命周期管理。启用/禁用、安装/卸载，都对运行中的应用零影响。

### 这个分层的关键洞见

很多人设计Skill系统时会做一个统一的"Skill接口"，把所有Skill（无论是内置还是外部）塞进同一个抽象中。这看起来很"干净"，但实际上是在**抹杀两种完全不同性质的实体的差异**。

好的设计不是"把所有东西统一"，而是**让不同性质的东西有不同的管理方式，但在使用层面提供统一体验**。这就是"在使用层面统一、在管理层面对齐差异"的思想。

## 三、@Tool注解的本质：给LLM看的API文档

Spring AI的 `@Tool(description="...")` 注解看起来很普通，但它背后是一个被大多数人忽视的事实：

**这不是注释。这是LLM的工具说明书。**

```java
@Tool(description = "计算数学表达式。支持+ - * / ** sqrt abs sin cos log exp，常量PI。" +
       "当用户问数学计算题时必须使用此工具，不要心算。")
```

这个description会经历以下旅程：
1. Spring AI扫描 `@Tool` 注解 → 提取方法签名和参数
2. 生成JSON Schema（兼容OpenAI Function Calling格式）
3. 注入到System Prompt中
4. LLM读取 → 和其他工具的Schema对比 → 选择最合适的工具
5. LLM根据参数Schema构造调用参数

所以"写好工具描述"的本质是**和LLM做API文档沟通**。如果你的文档不清晰，LLM就会选错工具。这和人类工程师看着糟糕的API文档会调错接口，是同一个道理。

### 一个关键实验

尝试用两个不同措辞的描述写同一个工具，观察Agent行为：

> 描述A："计算表达式"
> 描述B："计算数学表达式。支持+ - * /。当用户问数学题时必须使用此工具，不要自己心算。"

面对用户"3.14×256等于多少"，A描述的Agent有40%概率选择心算（不调用工具），B描述的Agent几乎100%调用工具。

这证明了一个重要的事实：**Agent开发中，Prompt Engineering和代码同等重要。** 工具描述是Prompt Engineering的一部分。

## 四、外部Skill的执行模型

外部Skill有三种类型：PROMPT、TOOL、MCP。三种不是随意分的，每种服务于一类场景。

### PROMPT类型：用知识改变行为

PROMPT类型的Skill本质是什么？是一个**可插拔的System Prompt片段**。

为什么需要这个？因为不同的任务需要Agent遵循不同的"规则"。比如：
- 代码审查任务 → 激活"代码审查规范"PROMPT → Agent按审查规则一项项检查
- 写周报任务 → 激活"周报模板"PROMPT → Agent按固定格式输出

PROMPT类型的巧妙之处在于：它不改变Agent的工具集，只改变Agent的"思考方式"。通过触发器关键词匹配（用户说"审查代码"→自动加载审查规则），实现了**上下文敏感的行为切换**。

### TOOL类型：外接"肢体"

HTTP API和Script两种执行器，本质上是给Agent两种不同的与外界交互的方式。

HTTP执行器的关键设计是 **`{param}` 占位符替换**。当LLM决定调用天气工具并传入 `city=北京`，执行器自动把URL `https://api.weather.com/{city}` 替换为 `https://api.weather.com/北京`。这个看似简单的字符串替换，实际上是**把LLM的结构化输出映射到外部API调用**的桥梁。

Script执行器的关键设计是**三层安全隔离**：
- 运行时白名单（只允许python3/bash/node）
- 临时文件隔离（不污染项目目录）
- 超时强制终止（防止死循环）

这三层不是随意加的。白名单防恶意，隔离防误操作，超时防资源耗尽。三者互补，构成完整的安全网。

## 五、工具注册的去重策略

一个容易被忽略的细节：当内置Skill和外部Skill都有"搜索"相关工具时，Agent会困惑。这就需要一个命名策略。

我们的做法是：**带前缀区分来源**。
- 内置工具：`calculator`、`translate`、`rollDice`
- 外部HTTP/Script工具：`ext_weather`、`ext_codexec`
- MCP工具：`mcp_read_file`、`mcp_search_web`

前缀不仅是为了避免冲突，更是给LLM一个"工具来源信号"——LLM可以根据前缀判断工具的可靠性。MCP工具来自标准化协议，可以"信任"；ext_工具来自用户自定义，需要"谨慎"。

## 六、从Skill系统到Agent生态的思考

回看Skill系统的设计，它其实在做一件很大的事：**为Agent建立一个能力市场的基础设施**。

今天的App Store有数百万App，是因为iOS/Android定义了标准化的App开发、分发、安装、权限管理体系。Agent的Skill系统也需要同样的标准化——不是你给每个Agent硬编码一套工具，而是有一个"Skill市场"，Agent需要什么就安装什么。

MCP协议（下一篇详谈）就是这个方向上的关键一步：它标准化了工具的"接口格式"。有了标准接口+可插拔Skill系统，Agent的能力就可以真正实现"按需安装"。

## 小结

Skill系统不只是"给Agent加几个工具"。它的核心设计问题是：

1. **如何不让工具多了之后Agent变笨？** —— 分层架构、精简Schema、命名策略
2. **如何让第三方能力安全地接入？** —— 外部Skill + 沙箱隔离
3. **如何让工具描述准确地引导LLM？** —— 把@Tool注解当"LLM沟通工具"而不是"代码注释"
4. **如何在统一使用体验的前提下管理异构能力？** —— 在注册层面统一、在执行层面各司其职

理解了这些，你就理解了为什么Agent工具系统的设计不是"加个接口"那么简单。
