# RAG的三次进化：从朴素检索到Agentic RAG

> RAG是解决LLM幻觉的主流方案。但大多数人对RAG的理解停留在"搜一次就答"。真正的生产级RAG经历了三次进化，每一次升级都解决了上一代的致命缺陷。

---

## 一、问题的起点：LLM为什么会产生幻觉？

LLM的幻觉不是bug，是设计特性。它的训练目标不是"输出正确信息"，而是"输出下一个最可能的token"。当被问到训练数据之外的问题时，它不会说"我不知道"——因为它的训练数据里没有"我不知道"对应的token序列——而是从概率分布中"挑选"一个听起来合理的答案。

RAG（Retrieval-Augmented Generation）的解决思路很直接：**不要让LLM凭记忆回答，先帮它查资料，让它基于资料回答。**

```
传统LLM: 问题 → 凭记忆回答（可能幻觉）
RAG:     问题 → 检索资料 → 让LLM基于资料回答（减少幻觉）
```

但这个看似简单的思路，在工程实现上经历了三次重要的进化。

## 二、Level 1：朴素RAG

```
文档 → Embedding模型 → 向量 → 向量数据库
用户提问 → Embedding模型 → 查询向量 → 相似度搜索 → Top-K文档块 → 拼入Prompt → LLM回答
```

### 这个方案的核心：语义搜索取代关键词搜索

传统搜索用TF-IDF或BM25，本质上是"出现相同单词的文档就是相关的"。但用户问"怎么提高系统性能？"时，相关文档可能用的是"性能优化"而不是"提高性能"——关键词匹配会漏掉。

向量搜索用Embedding模型把文本变成高维空间中的向量，用余弦相似度衡量"语义距离"。两个文本讨论相似的话题，即使用的词不同，它们的向量也会很接近。

### 朴素RAG的致命缺陷

这个方案能跑，但有两个问题：

1. **单次检索，质量不可控**：搜一次，拿到什么就用什么。如果查询向量化不准确、或者知识库中没有答案、或者检索结果不相关——LLM只能硬着头皮编。

2. **没有"我不确定"机制**：LLM拿到一份不相关的检索结果，它不知道这是错误的参考，还是会基于这些"幻觉材料"继续编。

## 三、Level 2：Agentic RAG

Agentic RAG的关键创新：**让Agent参与检索决策，而不仅仅是检索执行器。**

```
朴素RAG:    用户提问 → 检索 → 回答
Agentic RAG: 用户提问 → Agent判断 → 需要检索吗？
                                        ├── 不需要 → 直接回答
                                        └── 需要 → 生成检索query → 检索
                                                  → Agent评估检索质量
                                                  → 不好 → 改写query重搜
                                                  → 好 → 基于结果回答
```

### 为什么"检查检索质量"如此重要？

看一个例子。用户问："Spring AI的MCP支持怎么样？"

```
第一次检索 query = "Spring AI MCP支持"
  → 返回5条结果，平均相似度 0.45

Agent评估：
  - 相似度 < 0.5：结果不可靠
  - 决策：改写query重搜

第二次检索 query = "Spring AI MCP集成 使用方法 配置"
  → 返回5条结果，平均相似度 0.78

Agent评估：
  - 相似度 > 0.7：结果可靠
  - 决策：基于此回答
```

Agent为检索注入了"质量意识"。它不再盲目接受检索结果，而是检查结果质量，决定是否重试。这就是"Agentic"的含义——**不是机械执行，而是主动决策**。

### 相似度阈值的设计思考

设置什么阈值来判断"结果可用"？这没有标准答案，需要根据嵌入模型、知识库内容、应用场景调整。

一个经验法则：
- **> 0.8**：高相关性，结果可靠
- **0.5-0.8**：部分相关，需要考虑是否重搜
- **< 0.5**：低相关性，几乎确定需要重搜

但这个阈值也取决于嵌入模型的特性。BGE系列模型在中文上的相似度分布和OpenAI的text-embedding-3系列可能不同。所以**阈值应该是可调节的配置项，而不是硬编码常量**。

## 四、Level 3：多跳RAG + 重排序

这是Agentic RAG的高阶版本，解决的是"一个简单问题需要跨越多个文档来回答"的场景。

```
用户问："文档A提出的方案和文档B的方案有什么区别？"

单跳RAG：搜"方案区别" → 可能一条都找不到
多跳RAG：
  跳1：搜索"方案A的核心内容" → 从文档A获取相关块
  跳2：搜索"方案B的核心内容" → 从文档B获取相关块
  跳3：Agent对比两次检索结果 → 综合回答
```

多跳RAG的本质是**把复杂问题分解为多个子问题，每个子问题独立检索**。这是对Agent推理能力和检索能力的双重考验。

重排序（Reranking）是另一个重要升级。传统RAG只依赖向量相似度排序，但向量相似度和"实际相关性"不是完全等价的。Reranker是一个专门的模型（比如Cohere Rerank），在初次检索结果的基础上做二次排序，把真正相关的结果提到前面。

## 五、切片策略：RAG中最被低估的环节

几乎所有的RAG教程都在讲向量数据库和检索算法，但很少有人深入讨论**切片策略**——这恰恰是对最终效果影响最大的环节之一。

### 为什么切片这么重要？

向量检索的粒度是"块"（chunk）。如果一块包含太多内容，检索结果就不准确（"噪声"太多）。如果一块太小，语义不完整（"上下文"丢失）。

### 固定大小 vs 语义切片

最简单的策略是固定大小切片（比如每500字符一块），但这会切断语义。一个段落的开头和结尾可能分布在两个不同的chunk中，检索时可能只命中一半。

改进方案是**优先在自然边界切分**：段落边界（`\n\n`）> 行边界（`\n`）> 强行截断。这样每个chunk尽量保持语义完整性。

### 重叠的必要性

相邻chunk之间保留一定字符重叠（比如50字符），防止关键信息恰好落在chunk边界被切断。这不是"浪费"，而是冗余保证可靠性。

### 切片策略的未来：语义切片

更高阶的做法是"语义切片"——不用固定长度，而是根据语义相似度动态决定切分点。两个相邻句子的语义突然变化时，说明到了话题边界，应该在这里切分。但语义切片的计算成本比固定切片高一个数量级，需要权衡。

## 六、向量索引：HNSW为什么快？

pgvector支持两种索引：IVFFlat（聚类索引）和HNSW（图索引）。

### IVFFlat的原理

把向量聚成K个类，搜索时先找最近的类，再在类内搜索。问题：如果查询向量刚好在类边界上，最近的几个邻居可能分布在多个类中，聚类索引可能漏掉。这就是"边界效应"。

### HNSW的原理

构建一个多层的"邻居图"：每个向量节点连接到最近的m个邻居节点（m=16）。搜索时从上层（稀疏连接）快速定位到附近区域，然后在下层（密集连接）进行精细搜索。

HNSW的优势：**天然支持近似最近邻搜索，且不会漏掉边界附近的点**。代价是构建索引更慢，内存占用更大。

在我们的项目中，SQL搜索的核心是：

```sql
-- <=> 是pgvector的余弦距离运算符
-- 1 - (embedding <=> query) 是余弦相似度，范围0~1
SELECT content, 1 - (embedding <=> query_vector) AS similarity
FROM documents
WHERE 1 - (embedding <=> query_vector) > 0.6   -- 相似度阈值
ORDER BY embedding <=> query_vector
LIMIT 5;
```

## 七、从RAG到Agentic RAG的范式转变

最后总结一下这个范式转变的本质：

朴素RAG是**管道模式**（Pipeline）：输入→固定处理→输出。每步都是确定的，没有分支判断。

Agentic RAG是**决策模式**（Decision Loop）：输入→Agent判断→行动→评估→决定下一步。这是一个带反馈的循环，Agent在循环中主动做出决策。

管道的优势是简单可靠，决策的优势是灵活智能。Agentic RAG的挑战在于平衡这两者——引入智能决策的同时，保持系统的可控性和可预测性。

## 小结

RAG不是"加个向量库"这么简单。三次进化反映了三个关键认知：

1. **语义检索≠语义理解**：Embedding只是把文本变成数字，检索到的内容还需要Agent判断是否真的相关
2. **检索质量需要反馈循环**：Agentic RAG的核心不是更好的检索算法，而是"检索→评估→修正"的闭环
3. **切片策略比检索算法更影响效果**：决定"什么内容进入向量库"比决定"怎么从向量库找内容"更基础

理解这些，你就理解了为什么RAG的工程落地往往需要几个月的打磨，而不是几天的集成。
